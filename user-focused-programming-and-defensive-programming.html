Why You Should Use User Focused Programming vs Defensive Programming

Apr 3, 2014

Programmers program errors. That's a fact. So, you need to handle errors. Every major language has constructs for handling errors. There are also many guidelines such as defensive programming, and fail-fast programming. The problems are that everyone has a different definition, and we need to consider are our goals. We need to choose the method that best fits our goals. I want to examine the various definitions of error handling in different languages, and propose what should be done.

User Centered Programming is error handling with a goal. The goal is to give the user the best possible experience in the prescence of errors. A crashing program is a terrible user experience. Error handling has been around for so long that there is in danger of possessing the mentality of by programmers for programmers, otherwise known as "Meh, someone else can figure it out". Some old articles even recommend producing stack traces, and cryptic error codes to the user as the best way to handle errors. That's just horrid. Oh, how the times have changed! Even if you are actually developing software for other developers, we're users too! After all this, if we can make life easier for us, the developers, then life is good. Let's set our goals.

  • Application must not crash for the user. Ever.
  • Errors should be exposed to users in a friendly way if applicable.
  • All errors need a method of being known.

Now we need to achieve these goals.

Defensive Programming

In theory defensive programming basically means this.

Defend against the impossible, because the impossible will happen. [4]

In practice is means a whole smorgasbord of things. To me, defensive programming and fail-fast programming are complementary. Whatever the case, defensive programming will usually fall into one of two categories: handling function arguments, and using one-size fits all syntax. Although, defensive programming is also sometimes used to refer to not letting the program crash.

Handling function arguments

This is mostly what I think of when someone refers to defensive programming. It is one of the good parts. It just means a function should have correctly defined behaviour for all inputs. A simple contrived example.

private int currency;

// no defensive programming
public List<String> getFriendNames(User user) {
    List<User> users = user.getFriends();
    List<String> names = new ArrayList<String>();

    for (User user : users) {
        names.add(user.getName());
    }
}

In this example there is no defensive programming. All sorts of stuff could happen. null could be passed in and we'd get the infamous NullPointerException. users could be null, or the name might be null. Maybe it might look like this instead.

private int currency;

// better
public List<String> getFriendNames(User user) throws InvalidFriendListException {
    if (user != null) {
        List<User> users = user.getFriends();
        List<String> names = new ArrayList<String>();

        if (users != null) {
            for (User user : users) {
                String name = user.getName();
                names.add(name == null ? "" : name);
            } 
        } else {
            throw new InvalidFriendListException();
        }

    } else {
        throw new InvalidFriendListException();
    }
}

This code prevents undefined behaviour. If an input comes in that we did not expect then the caller should handle it. Sometimes people write about how terrible defensive progamming is. This is generally when someone does something like this.

private int currency;

// bad
public List<String> getFriendNames(User user) {
    List<String> names = new ArrayList<String>();

    if (user != null) {
        List<User> users = user.getFriends();

        if (users != null) {
            for (User user : users) {
                String name = user.getName();
                names.add(name);
            } 
        }
    }

    return names;
}

Here people complain about bugs being covered up. This is valid. We could get back an empty list and have no idea if something went wrong or if this user just does not have any friends. The second example was better. We coded defensively in anticipation of errors, and someone is still going to be notified about the bug.

However, defensive programming in this way can be cumbersome and lead to less readability. Every function ends up with this extra defense around the core logic, and defining all your custom exceptions leads to bloat. Yuck. Clojure can help us out a bit.

;; user is a map
;; ex.
;; {:friends [{:name "Steven"}
;;            {:name "Jake"}
;;            {:name nil}]}
(defn get-friend-names [user]
    (if-let [result (remove nil? (map :name (:friends user)))]
        result
        (throw (InvalidFriendListException.))))

Many functions in Clojure safely accept nil arguments. In this case if user is nil then (:friends user) is also nil. Mapping :name over nil is nil, and removing all nil elements from nil is nil. That's kind of sweet.

However, null pointer exceptions still happen. For example, with number ops or java interop. In addition, the if branches are still used frequently for defensive programming. Maybe it doesn't make sense to execute code with a nil value, possibly because it will crash some arithmetic. You'd still use nil? to defensively check values.

Erlang is one step better. In Erlang, your functions use pattern matching to define acceptable values. Anything unacceptable automatically throws an exception. 

-module(test).
-export([get_friend_names/1]).

get_friend_names({user, {friends, [{name, Name}|T]}}) ->
    [Name | get_friend_names({user, {friends, T}})];
get_friend_names({user, {friends, []}}) -> 
    [].

%% add me to throw a custom error meant to be caught
%% get_friend_names(X) -> throw({invalid_friends_list, X}).

That's it. The function recursively builds up a list, getting the name of one friend each call. The base case is when we have no more friends. In that case, we return an empty list. If any input doesn't match our pattern, an exception is thrown. This is what I mean when I say Erlang gives you defensive programming for free. Some people cringe at the exception that gets thrown here, possibly because it's very common. You need an exception here though. In any other language if you're not throwing an exception you're either hiding bugs or not coding defensively.

Using one-size fits all syntax

I don't see any downsides to this. The main premise is that you write your syntax in a way that will avoid errors if the code changes later. For example, if you want to add an expression to an if statement without curly braces you might forget to add the curly braces.

if (x == 4) {
    x++;
}

// instead of

if (x == 4)
    x++;

// because

if (x == 4)
    x++;
    x *= 5;

A separate example in JavaScript.

var obj = {a = 4
          ,b = 5
          ,c = 10
          ,d = []}; // added this line later

// vs

var obj = {a: 4,
           b: 5,
           c: 10 // doh! missing a comma
           d: []};

Fail-fast Programming

Fail-fast programming is probably best described by someone else's quote.

whereby an application or system service terminates itself immediately upon encountering an error

At first this may not make sense. There tends to be this confusion that defensive programming hides bugs, but your program never crashes. Where as fail-fast programming exposes bugs quickly, but at the expense of your program crashing immediately. Maybe this is the case in traditional languages. When programs usually run in just a few threads, or a large portion of the execution is in a single thread a failure might mean a crash. However, fail-fast can work very well in the right context.

Fail-fast is not mutually exclusive from defensive programming. We've already seen that Erlang supports defensive programming. Erlang also embraces fail-fast programming. That does not mean that Erlang programs are meant to crash. In fact, one of its selling points is that programs are meant to last forever. This works because Erlang encourages creating programs with many processes. If one process crashes another can clean up the errors. It's like an inter-process try catch.

This does not mean that in Erlang every time any error occurs a process dies. You should be reasonable. Erlang has a try catch clause just like other languages. Small errors can be handled in the same way that Java handles small errors. The difference is in bigger errors. Imagine that my User object all of a sudden does not have a friends list. It's not empty, just gone. In Java you need to try hard to fix the state. Perhaps you re-initialize the friends list and pull from the database. In Erlang maybe you just restart the process that represents that user.

The implications are that in Erlang a lot of error handling code can be separated from your logic instead of being coupled, and many times it's much easier to fix the state by just starting over. In addition, error handling can easily be distributed! One process can transparently monitor another process even when they are on different machines. This opens up the door for some interesting software architectures.

Reaching the top

We have validated our function inputs. If anything goes wrong we throw exceptions to be handled someplace else. What to do we do when they reach the top of the stack? We never let our app crash. That is a terrible experience.

Worst case scenario we display a human readable error to the user. Perhaps a friendly red dialog box fades in. If the user is not expecting a response we should log the error to a persistent source. Sending an email works. There are services that will keep track of your errors. On Android you can log your errors and view them in your application dashboard. This ensures that you are aware of all errors.

Conclusion

We had three goals. By coding defensively we made errors known when they occur, and known what errors we should expect. Uncaught errors should be caught at the top of the stack, or crash the process if it can be restarted. Either way, we log our errors so that none go missed. This allows us to fix our bugs and improve software quality.

If the user is expecting an action to happen a friendly error should be given so they know what has happened. By catching our errors, or monitoring processes we can make sure that our application never crashes. However, none of these tactics are a substitute for testing. Developer's should test, and there should be a goot test suite to mitigate as many errors as possible before your user's start using the product.
